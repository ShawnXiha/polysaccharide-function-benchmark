## Key Observations

- Pipeline artifacts should exist before the first executable model run.
- Dataset and split generation are part of Stage 1 infrastructure, not optional cleanup.
- Dependent ETL commands should be run sequentially once file outputs are chained.

## Lessons Learned

- [Reusable] Start with fixed stage folders and trajectory logs, not ad hoc notebooks.
- [Reusable] Fix the dataset contract before baseline code, or every early model result becomes non-comparable.
- [Reusable] When one script consumes another script's output file, do not parallelize them.
- [Reusable] Stage 1 success is about executability and determinism first, not headline accuracy.
- [Reusable] Add a learned classical baseline before moving to sequence or graph models.
- [Reusable] Validate grouped split formats with a simple baseline before adding heavier models.
- [Reusable] Complete a baseline family across the target splits before branching into a new model family.
- [Reusable] Add the weakest valid member of a family before the heavier version.
- [Reusable] For graph-level multi-label tasks in PyG, keep label tensors graph-level rather than node-level.
- [Reusable] A local scaffold can complete Stage 1 engineering checks before real-data ingestion begins.
- [Reusable] After real-data ingestion works, verify one cheap baseline before scaling back up to the full baseline suite.
- [Reusable] A readiness checker prevents premature transition into tuning on template-derived data.
- [Reusable] When you stand up a site-native crawler, force it to emit the same CSV schema as the manual export path so dataset builders and baselines do not fork.
- [Reusable] A source-shift run can be engineering-valid while still being scientifically weak if the held-out source is labeled under a different semantic regime.
